# hardware-encryption-test

Минимальное консольное приложение на Python для проверки PKCS#11 токена.

Что делает:
- печатает приветствие с версией приложения
- просит путь к PKCS#11 библиотеке
- если путь пустой, пытается загрузить библиотеку из директории приложения:
  - Windows: `rtpkcs11ecp.dll`
  - Linux: `librtpkcs11ecp.so`
  - macOS: `librtpkcs11ecp.dylib`
- загружает библиотеку и выводит общую информацию
- находит первый слот с вставленным токеном и выводит информацию о токене
- показывает TUI-меню действий
  1. найти все ключевые пары
  2. сгенерировать ключевую пару
  3. удалить ключевую пару по номеру
  4. подписать файл
  5. шифровать 500 килобайт данных
- при генерации предлагает выбор типа пары:
  - `0` по умолчанию, `ГОСТ Р 34.10-2012(256)`
  - `1`, `ГОСТ Р 34.10-2012(512)`
  - `2`, `RSA-2048`
- для генерации спрашивает `cka_id` и `cka_label`
- для поиска показывает все найденные пары с нумерацией с нуля
- для удаления и подписи используется номер пары, по умолчанию `0`
- для подписи приложение спрашивает путь к файлу и количество подписаний в диапазоне `1..10000`
- для шифрования приложение спрашивает путь к файлу и количество шифрований в диапазоне `1..10000`
- для шифрования приложение спрашивает режим:
  - `0` по умолчанию, программное шифрование, ключ создаётся через `C_GenerateKey` как `CKA_TOKEN=FALSE`
  - `1`, аппаратное шифрование, ключ создаётся через `C_GenerateKey` как `CKA_TOKEN=TRUE`
- вместо выбора ключевой пары для шифрования приложение предлагает выбор алгоритма:
  - `0` по умолчанию, `Магма`
  - `1`, `Кузнечик`
  - `2`, `ГОСТ 28147-89 (режим гаммирования)`
- для RSA подпись выполняется через `CKM_SHA256_RSA_PKCS`
- для ГОСТ подпись выполняется максимально близко к примерам Rutoken SDK:
  - `ГОСТ Р 34.10-2012(256)` через `CKM_GOSTR3410_WITH_GOSTR3411_12_256`
  - `ГОСТ Р 34.10-2012(512)` через `CKM_GOSTR3410_WITH_GOSTR3411_12_512`
  - при генерации в шаблоны ключей добавляются `CKA_GOSTR3410_PARAMS` и `CKA_GOSTR3411_PARAMS` с OID из SDK Rutoken
  - при подписи ГОСТ-ключом приложение спрашивает режим хеширования: `0` по умолчанию, программное хеширование библиотекой Rutoken; `1`, аппаратное хеширование внутри токена
- для демонстрации симметричного шифрования используется прямое создание секретного ключа через `C_GenerateKey`, без `VKO/KEK/CEK`
  - в меню доступен пункт `5) шифровать 500 килобайт данных`
  - после выбора режима приложение создаёт временный секретный ключ нужного алгоритма и использует его в `C_EncryptInit/C_Encrypt`
  - для `ГОСТ 28147-89` ключ создаётся с `CKA_GOST28147_PARAMS`
  - для `Магма` и `Кузнечик` используются механизмы `CTR_ACPKM`, параметры и буферы выставлены под реальные размеры алгоритмов
  - после завершения всех циклов шифрования секретный ключ удаляется через `C_DestroyObject`
  - в циклах подписи и шифрования убраны лишние предварительные вызовы `C_Sign(..., NULL, ...)` и `C_Encrypt(..., NULL, ...)`, чтобы не делать лишних обращений к токену на каждом прогоне
  - в отчёте показываются режим, `CKA_TOKEN`, алгоритм, механизмы, параметры последнего шифрования и статистика по времени
- если путь к файлу не введён, приложение пробует взять встроенный тестовый файл `lorem-500kb.txt` рядом с бинарником, а при запуске из исходников, из `testdata/lorem-500kb.txt`
- если во время работы пункта меню возникает ошибка, приложение показывает текст ошибки и возвращается в TUI-меню, а не завершает процесс

Важно:
- приложение состоит из одного файла: `app.py`
- PKCS#11 библиотека не встраивается в исполняемый файл и должна лежать отдельно
- для операций генерации, удаления и подписи нужен PIN токена
- GitHub Actions собирает артефакты для Windows x64, Linux x64 и macOS
- в артефактах исполняемый файл и `lorem-500kb.txt` лежат в одной папке
